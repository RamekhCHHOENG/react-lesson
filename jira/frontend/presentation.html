Routing & Code Splitting

React Router v7 with lazy loading and protected routes

// Lazy loading — pages only load when visited const BoardPage = lazy(() => import("@/features/board/BoardPage") ); const BacklogPage = lazy(() => import("@/features/backlog/BacklogPage") ); // Suspense wrapper with skeleton fallback function LazyPage({ children }) { return ( <Suspense fallback={<PageSkeleton />}> {children} </Suspense> ); } // Route tree with guards createBrowserRouter([ { // Public (login, register) element: <PublicRoute />, children: [{ element: <AuthLayout />, ... }] }, { // Protected (dashboard) element: <ProtectedRoute />, children: [{ element: <DashboardLayout />, children: [ { path: "/", element: <ForYouPage /> }, { path: "/board", element: <BoardPage /> }, { path: "/backlog", element: <BacklogPage /> }, { path: "/summary", element: <SummaryPage /> }, ] }] } ]);

Key React Concepts

lazy()
Dynamically imports components — only loads the JS bundle when the user navigates to that page. Reduces initial load time.
Suspense
Shows a fallback (skeleton loader) while the lazy component is loading. Built-in React feature.
ProtectedRoute
Checks authentication state. Redirects to /login if not authenticated. Uses Navigate component.
Outlet
Renders nested child routes inside layout components. Enables shared sidebar + nav.

State Management: Context API + useReducer

Auth state managed with the reducer pattern

// store/auth-store.tsx // 1. Define state shape interface AuthState { user: User | null; token: string | null; isAuthenticated: boolean; isLoading: boolean; } // 2. Define action types type AuthAction = | { type: "LOGIN_START" } | { type: "LOGIN_SUCCESS"; payload: { user; token } } | { type: "LOGIN_FAILURE"; payload: string } | { type: "LOGOUT" }; // 3. Reducer function (pure!) function authReducer(state, action): AuthState { switch (action.type) { case "LOGIN_SUCCESS": return { ...state, user: action.payload.user, token: action.payload.token, isAuthenticated: true, isLoading: false, }; case "LOGOUT": return { ...initialState }; } } // 4. Provider component function AuthProvider({ children }) { const [state, dispatch] = useReducer( authReducer, initialState ); // ... login, logout methods return ( <AuthContext.Provider value={value}> {children} </AuthContext.Provider> ); }

Why useReducer over useState?

  • Predictable transitions — actions describe what happened
  • Complex state — multiple related fields change together
  • Pure function — reducer is easy to test
  • Centralized logic — all state changes in one place

React Hooks Used

Built-in hooks powering the application

useState
Local component state — form inputs, toggle booleans, search queries, sidebar collapsed state. The most fundamental hook.
const [query, setQuery] = useState(""); const [open, setOpen] = useState(false);
useEffect
Side effects — apply theme to DOM, restore session from localStorage, attach keyboard listeners, WebSocket setup.
useEffect(() => { root.classList.add(theme); }, [theme]);
useContext
Access context values — auth state, selected project, theme. Avoids prop drilling across component tree.
const { user, token } = useContext(AuthContext);
useReducer
Complex state with actions — auth flow (LOGIN_START, LOGIN_SUCCESS, LOGOUT). Predictable state transitions.
const [state, dispatch] = useReducer(authReducer, init);
useMemo
Expensive computation cache — filtering task lists, computing chart data. Only recalculates when dependencies change.
const filtered = useMemo(() => tasks.filter(t => match(t, q)), [tasks, q]);
useCallback
Memoize functions — drag-end handler for board. Prevents unnecessary re-renders of child components.
const onDragEnd = useCallback( (result) => { /* ... */ }, [project, updateTask]);

Data Fetching: React Query

Server state management with caching, mutations, and optimistic updates

Reading Data — useQuery

// hooks/useTasks.ts export function useTasks(projectId) { return useQuery<Task[]>({ queryKey: ["tasks", projectId], queryFn: () => api.get( `/api/projects/${projectId}` ), enabled: !!projectId, // conditional staleTime: 60_000, // 1 min cache }); }

Writing Data — useMutation

export function useCreateTask() { const qc = useQueryClient(); return useMutation({ mutationFn: ({ projectId, data }) => api.post(`/api/projects/${projectId}/tasks`, data), onSuccess: () => { // Invalidate cache ⟶ triggers refetch qc.invalidateQueries({ queryKey: ["projects"] }); toast.success("Task created"); }, }); }

How React Query Works

1. Component calls useQuery(["tasks", id])
↓
2. React Query checks cache — is data fresh?
↓
3. If stale → calls queryFn (API request)
↓
4. Updates cache → re-renders component

Benefits

  • Auto caching — no duplicate requests
  • Background refresh — stale data updated
  • Loading/error states — built-in booleans
  • Optimistic UI — instant feedback

Optimistic Updates

Update the UI instantly, before the server responds

export function useUpdateTask() { const qc = useQueryClient(); return useMutation({ mutationFn: ({ projectId, taskId, data }) => api.put(`/api/projects/${projectId}/tasks/${taskId}`, data), // STEP 1: Before API call — update UI immediately onMutate: async ({ projectId, taskId, data }) => { await qc.cancelQueries({ queryKey: ["projects"] }); // Cancel background fetches const previousProjects = qc.getQueryData(["projects"]); // Snapshot for rollback qc.setQueryData(["projects"], (old) => // Optimistically update cache old.map(proj => proj.id !== projectId ? proj : { ...proj, tasks: proj.tasks.map(task => task.id === taskId ? { ...task, ...data } : task ), }) ); return { previousProjects }; // Return snapshot }, // STEP 2: On error — rollback to snapshot onError: (err, __, context) => { if (context?.previousProjects) { qc.setQueryData(["projects"], context.previousProjects); // Restore old data } toast.error(err.message); }, // STEP 3: Always — sync with server onSettled: () => { qc.invalidateQueries({ queryKey: ["projects"] }); // Refetch latest data }, }); }

Controlled Forms

React controlled component pattern for form handling

// TaskFormDrawer.tsx — Create / Edit const [formData, setFormData] = useState({ title: "", description: "", issue_type: "task", status: "todo", priority: "medium", }); const [errors, setErrors] = useState({}); // Validation function validate() { const newErrors = {}; if (!formData.title.trim()) newErrors.title = "Summary is required"; setErrors(newErrors); return Object.keys(newErrors).length === 0; } // Submit handler async function handleSubmit(e) { e.preventDefault(); if (!validate()) return; if (isEditing) { await updateTask.mutateAsync({ projectId, taskId: task.id, data: formData }); } else { await createTask.mutateAsync({ projectId, data: formData }); } } // Controlled input <Input value={formData.title} onChange={(e) => setFormData(prev => ({ ...prev, title: e.target.value }) )} />

API Service Layer

Centralized HTTP client with JWT authentication

// services/api.ts — Generic API Client class ApiClient { private baseUrl: string; // Generic GET with type parameter async get<T>(path: string): Promise<T> { const res = await fetch( `${this.baseUrl}${path}`, { headers: { Authorization: `Bearer ${ApiClient.getStoredToken()}` }, } ); if (!res.ok) throw new ApiRequestError( res.status, "Request failed" ); const json = await res.json(); return json.data; } // POST, PUT, DELETE follow same pattern async post<T>(path, body?): Promise<T> { ... } async put<T>(path, body): Promise<T> { ... } async del(path): Promise<void> { ... } // JWT Token management static setToken(token: string): void { localStorage.setItem("jira_token", token); } static clearToken(): void { localStorage.removeItem("jira_token"); } } export const api = new ApiClient(BASE_URL);

Why a Service Layer?

  • Separation of concerns — components don't know about fetch/HTTP
  • DRY — auth header added once, not in every component
  • TypeScript generics — type-safe responses
  • Error handling — centralized error parsing
  • Easy to test — mock the service, not fetch

Usage in Hooks

// Clean consumption in custom hooks const projects = api.get<Project[]>("/api/projects"); const newTask = api.post<Task>( `/api/projects/${id}/tasks`, formData ); const updated = api.put<Task>( `/api/projects/${id}/tasks/${tid}`, { status: "done" } );

Error Boundary & Theme Provider

Class component pattern for error catching + Context for theming

Error Boundary (Class Component)

class ErrorBoundary extends Component { state = { hasError: false, error: null }; // Catch errors during rendering static getDerivedStateFromError(error) { return { hasError: true, error }; } // Log error details componentDidCatch(error, info) { console.error("Caught:", error, info); } render() { if (this.state.hasError) { return ( <div> <AlertTriangle /> <h2>Something went wrong</h2> <Button onClick={this.retry}> Try Again </Button> </div> ); } return this.props.children; } }

Theme Provider (Context)

function ThemeProvider({ children }) { const [theme, setTheme] = useState(() => localStorage.getItem("theme") || "dark" ); // Apply theme class to <html> useEffect(() => { const root = document.documentElement; root.classList.remove("light", "dark"); root.classList.add(theme); }, [theme]); return ( <ThemeContext.Provider value={{ theme, setTheme }}> {children} </ThemeContext.Provider> ); } // Custom hook for consuming export const useTheme = () => useContext(ThemeProviderContext);
💡 Error Boundaries are the only React pattern that still requires a class component — no hook equivalent exists.
